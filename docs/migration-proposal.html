🧭 Guiding Principle

The crew vault is for agents. Port what helps agents understand Ben's thinking, preferences, decisions, and patterns. Leave what's personal, private, raw, or time-bound. Synthesize before copying — don't dump raw notes into the new vault.
Port Rule
If an agent needs to know it to do useful work for Ben → port. If it's Ben's personal reflection, health data, journal, or raw reference dump → stays.
Synthesize Rule
Before copying a note verbatim, check if the insight can be consolidated with related notes. The crew vault should have fewer, denser notes than the source.
Frontmatter Rule
Every ported note gets the crew vault frontmatter standard: date, type, tags, ai-first: true. Add ## For future agents preamble. Add confidence and source where known.

⚡ Execution Plan

Phase 1 — Direct Ports (Scribe-driven)
For each of the ~30 direct-port notes:
1. Read the source from Vault-v2 via ext.read
2. Add crew vault frontmatter
3. Add ## For future agents preamble (2-3 sentences: what, why, staleness)
4. Write into appropriate crew vault folder
5. Add wikilinks to related notes

Dry-run first, review the proposed files, then execute.
Phase 2 — Synthesized Ports (Seeker + Scribe)
For each of the 6 synthesis targets:
1. Seeker reads all source material from Vault-v2
2. Seeker identifies cross-cutting insights and patterns
3. Scribe writes a single condensed note with provenance
4. Frontmatter includes origin_names listing all source files

These are the most valuable notes — they capture patterns, not just data.
Phase 3 — Crew-Native Pages (Scribe)
Written fresh, informed by the assessment:
1. ben-system-map.md — every system in Ben's infrastructure
2. ben-memory-architecture.md — how vaults, mem0, and agent-brain relate
3. ben-projects-index.md — what's active, what's paused
4. ben-conventions.md — consolidated agent behavioral rules

These fill gaps the assessment identified. They make the crew vault self-contained.
